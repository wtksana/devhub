# 数据库 B 版第二批：表数据编辑设计

## 背景

数据库 B 版第一批已经支持双击表进入只读表数据浏览模式，包含分页、单列排序、简单 SQL 条件筛选和刷新。第二批在这个浏览模式上增加可控编辑能力，满足日常直接修正少量数据的场景。

编辑能力仍然保持保守边界：只允许有主键的普通表编辑，保存前必须确认；无主键表和视图保持只读。后续实现已经扩展到新增行、删除行、键盘编辑、拖选复制和 TSV 批量粘贴，但粘贴不会自动新增行，不会覆盖主键或只读列，也不会越界写入。

## 目标

- 有主键的普通表支持编辑已有行的单元格。
- 有主键的普通表支持新增行和删除行。
- 没有主键的表保持只读，并显示只读原因。
- 视图保持只读。
- 双击单元格进入编辑状态。
- 支持点击选中单元格、键盘移动、拖选范围和复制选中内容。
- 支持从 Excel / DataGrip 等工具复制 TSV 后批量粘贴到当前已加载范围内的可编辑单元格。
- 失去焦点不自动保存，只保留本地未保存更改。
- 顶部显示 `保存更改` 和 `放弃更改`。
- 保存前弹确认，显示将更新的行数和字段数。
- 保存成功后刷新当前页。
- MySQL 和 PostgreSQL 都支持。

## 不做范围

- 大文本专用编辑器。
- JSON 专用编辑器。
- 二进制字段编辑。
- 粘贴时自动新增行。
- 粘贴覆盖主键或只读列。
- 粘贴跨越当前已加载表格范围。
- 乐观锁版本字段。
- 事务跨页保存。
- 表结构管理。
- 视图更新。

## 推荐方案

采用“表数据分页结果 + 主键元数据 + 本地脏单元格”的方案。

- 后端加载表数据时返回主键列列表。
- 前端根据主键列判断当前表是否可编辑。
- 用户双击非主键单元格后进入输入态。
- 输入完成只更新前端本地脏数据，不调用后端。
- 用户点击保存时，前端把每一行的主键值和变更字段提交给后端。
- 后端根据主键构造 `WHERE`，只更新被修改的字段。

这个方案的优点是实现边界清晰，不需要前端猜测主键，也不会因为排序、筛选或分页导致无法定位行。缺点是第一版依赖主键；没有主键的表不能编辑，但这符合安全优先的原则。

## 数据模型

### 表数据响应

现有 `DatabaseTablePageResult` 增加两个字段：

- `primary_key_columns: string[]`
- `editable: boolean`

规则：

- 普通表且存在至少一个主键列：`editable = true`
- 普通表但无主键：`editable = false`
- 视图：`editable = false`

前端仍使用 `columns` 和 `rows` 渲染表格。`primary_key_columns` 用于：

- 判断行标识。
- 禁止编辑主键列。
- 构造保存请求中的主键值。

### 更新请求

新增 Tauri command：

```rust
update_database_table_rows(request) -> DatabaseTableUpdateResult
```

请求字段：

- `connection_id`
- `database`
- `table`
- `rows`

每个 `rows` 条目包含：

- `primary_key_values`: `{ column: value }`
- `changes`: `{ column: value }`

响应字段：

- `updated_rows`
- `updated_fields`
- `duration_ms`

第一版 `value` 使用字符串协议：

- 空字符串保存为空字符串。
- `NULL` 值第一版不通过输入框直接编辑；原值是 `NULL` 时显示为 `NULL`，用户修改后按文本保存。
- 数字、布尔、日期都作为输入文本传给数据库驱动，由数据库类型转换；转换失败时返回后端 SQL 错误。

## 后端设计

### 主键识别

MySQL：

```sql
select column_name
from information_schema.key_column_usage
where table_schema = ?
  and table_name = ?
  and constraint_name = 'PRIMARY'
order by ordinal_position
```

PostgreSQL：

```sql
select kcu.column_name
from information_schema.table_constraints tc
join information_schema.key_column_usage kcu
  on tc.constraint_name = kcu.constraint_name
 and tc.table_schema = kcu.table_schema
where tc.table_schema = $1
  and tc.table_name = $2
  and tc.constraint_type = 'PRIMARY KEY'
order by kcu.ordinal_position
```

表名和列名继续由后端使用现有标识符引用函数处理。

### 更新 SQL

MySQL 示例：

```sql
update `table`
set `name` = ?
where `id` = ?
```

PostgreSQL 示例：

```sql
update "schema"."table"
set "name" = $1
where "id" = $2
```

复合主键使用多个 `AND` 条件。

安全边界：

- 表名、字段名、主键列必须由后端引用。
- 后端会重新查询主键列，不能完全相信前端传入。
- `changes` 中不能包含主键列。
- `changes` 为空时拒绝更新。
- 没有主键时拒绝更新。
- 每行更新影响行数不是 1 时返回错误，避免误更新或行已不存在。

## 前端设计

### 工具栏

表数据浏览工具栏在现有控件基础上增加：

```text
保存更改    放弃更改    未保存 x 行 / y 字段
```

只有存在未保存更改时按钮可用。

只读时显示简短提示：

- `当前表没有主键，表数据只读。`
- `视图暂不支持编辑。`

### 单元格编辑

- 单击单元格只选中，不立即进入输入态。
- 双击可编辑单元格进入输入态。
- 主键列不可编辑。
- 只读表不可编辑。
- 编辑输入按 Enter 确认当前单元格。
- 按 Esc 取消当前单元格编辑，恢复进入编辑前的值。
- 失焦只确认当前单元格为本地更改，不触发保存。
- 已修改单元格用主题适配的背景色标记。
- 支持方向键、Tab / Shift+Tab、Enter / F2、Esc、Delete / Backspace 和 Ctrl+C 等表格键盘操作。
- 支持拖选范围，并在右键菜单中复制选中单元格。
- 支持 Ctrl+V / Cmd+V 粘贴 TSV 内容，从当前选中单元格开始写入可编辑单元格。
- 粘贴值和原值一致时会移除对应单元格的本地脏标记。

### 保存流程

用户点击 `保存更改`：

1. 弹确认窗口。
2. 显示 `确认保存 x 行 y 个字段的更改？`
3. 用户确认后调用 `update_database_table_rows`。
4. 保存成功后清空本地脏数据并刷新当前页。
5. 保存失败时保留本地脏数据并显示错误。

用户点击 `放弃更改`：

1. 如果存在未保存更改，弹确认窗口。
2. 确认后清空本地脏数据。
3. 不刷新数据。

### 分页、排序、筛选切换

如果存在未保存更改，用户执行这些动作前需要确认：

- 上一页 / 下一页。
- 修改页码。
- 修改每页条数。
- 点击排序。
- 修改筛选。
- 刷新。
- 切换到其他表。

第一版处理策略：弹出确认，提示 `当前有未保存更改，继续操作会放弃这些更改。`。确认后清空本地更改并继续操作；取消则停留当前状态。

## 错误处理

- 没有主键：前端显示只读提示，后端更新命令也拒绝。
- 更新影响行数不是 1：返回错误并保留本地更改。
- 类型转换失败：显示数据库错误并保留本地更改。
- 连接失败：显示错误并保留本地更改。
- 保存过程中禁用保存按钮，避免重复提交。

## 测试策略

### Rust 测试

- MySQL 主键查询 SQL。
- PostgreSQL 主键查询 SQL。
- 有主键时更新 SQL 生成。
- 复合主键更新 SQL 生成。
- 无主键拒绝更新。
- 更新字段包含主键时拒绝更新。
- 空变更拒绝更新。

### 前端测试

- 有主键表显示可编辑状态。
- 无主键表显示只读提示。
- 双击普通单元格进入编辑。
- 主键列不可编辑。
- 编辑后显示未保存统计。
- 保存前弹确认并调用后端。
- 保存成功后刷新当前页。
- 放弃更改清空脏状态。
- 有未保存更改时翻页 / 排序 / 筛选需要确认。

## 验收标准

- 双击有主键表的普通单元格可以编辑。
- 修改后不会自动保存。
- 保存前必须确认。
- 保存请求只包含主键值和改过的字段。
- 保存成功后当前页刷新。
- 无主键表不能编辑并有明确提示。
- 现有表数据分页、排序、筛选继续可用。
